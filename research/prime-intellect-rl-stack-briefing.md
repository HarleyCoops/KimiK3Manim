## For: RL pipeline training an LLM to write Manim code with verifiable rewards

## 1. PRIME-RL — the training framework

**What it is.** `prime-rl` is Prime Intellect's open-source (Apache 2.0), *async-only* large-scale RL/post-training framework. Repo: https://github.com/PrimeIntellect-ai/prime-rl. Version on `main` at fetch time: **0.7.0** (`pyproject.toml`), Python **3.12**, torch ≥2.9, **vLLM 0.24.0** (pinned wheel), transformers **5.6.2** (hard pin), `verifiers[harbor]>=0.2.0` (editable git submodule), torchtitan pinned to a git rev. There are no traditional semver releases; you track `main`.

**Architecture — three cooperating processes** (https://docs.primeintellect.ai/prime-rl, tech report §2.1 https://arxiv.org/abs/2512.16144):
- **Inference**: one or more standard vLLM servers, extended with `/update_weights` (hot-swap policy) and `/reload_weights`. The orchestrator drives rollouts through a token-in `/v1/generate` route via Prime's `renderers` package (no re-tokenization across turns); OpenAI-compatible chat routes are also exposed.
- **Orchestrator**: lightweight CPU/asyncio process. Owns the data plane across many `verifiers` envs (each env runs in isolated subprocess pools), drives multi-turn rollouts, computes advantages, packs rollouts into training batches, relays weights trainer→inference.
- **Trainer**: **FSDP2** process group, torchtitan-inspired, works with any HF causal LM (`impl="auto"`). Ships optimized custom modeling (expert parallelism via DeepEP kernels, context parallelism, FP8 on Hopper+, selective activation checkpointing) for GLM-4.5/GLM-5 MoE, Qwen3/Qwen3.5 (dense, MoE, VL), MiniMax-M2, Nemotron-H, GPT-OSS, Trinity, INTELLECT-3. LoRA and multi-tenant LoRA supported.
- **Transports**: trainer↔orchestrator rollouts via local filesystem by default (ZMQ for multi-host); weight broadcast via NCCL in-memory (filesystem fallback when LoRA or no inference server).

**Async design** (this is the core differentiator — "async-only"): trainer and inference run disaggregated on disjoint GPUs, always ≥1 step overlapped; at step *n* inference samples from policy π_{n−1}. **Continuous batching** keeps a saturated pool of in-flight rollouts (finished groups are immediately replaced — kills the "slowest rollout" bottleneck). **In-flight weight updates** interrupt generation to hot-load new weights; a single trajectory may span multiple policy versions, and rollouts generated by more than `max_off_policy_steps` policies are discarded to bound staleness. A **multi-client orchestrator** treats each inference node as an independent server with round-robin routing — linear throughput scaling where vLLM's own DP plateaus. Importance-ratio correction lives in the loss (below).

**Algorithms** (new first-class layer, July 2026 — https://www.primeintellect.ai/blog/algorithms-layer and https://docs.primeintellect.ai/prime-rl/algorithms.md): six built-ins under `[orchestrator.algo] type = "grpo" | "max_rl" | "opd" | "opsd" | "sft" | "echo"`, **resolvable per-environment** (GRPO on one env, ECHO on another, in one run). GRPO default advantage is DR-GRPO (reward minus group mean, no std normalization) with optional `length_penalty type="linear"`. The default RL loss is **DPPO + a KL regularizer "similar to Kimi-K2.5"**: `min(importance_ratio, δ)·Â` with one-sided clamp, plus `kl_tau` (default 1e-3) × squared log importance-ratio penalty; knobs `dppo_mask_low/high` (0.2/0.2), `adv_tau`. PPO-clip or anything else via `[trainer.loss] type="custom" import_path="my_module.my_loss"` (per-sequence function over `LossInputs`). Teachers/reference models are *external* OpenAI-compatible endpoints declared inline (e.g. `[orchestrator.algo.teacher]` for OPD) — prime-rl only ever serves the trainable policy. Built-in rollout **filters**: `gibberish`, `repetition`, `zero_advantage` (post-batch enforced, pre-batch monitor by default).

**Config system**: `pydantic-config` — TOML files passed with `@` (left-to-right deep-merge), CLI dotted kebab-case overrides, `--dry-run` to inspect resolved per-process configs. Environments are arrays of tables: `[[orchestrator.train.env]] id = "env-id"`, `args = {...}` (forwarded verbatim to `load_environment(**args)`), `ratio` for mixing weights; `group_size` per env (renamed from `rollouts_per_example`; old name still accepted). Launch everything with one command:
```
uv run rl --trainer @ configs/x/train.toml --orchestrator @ configs/x/orch.toml --inference @ configs/x/infer.toml
# or single-file: uv run rl @ examples/reverse_text/rl.toml
```
Checkpointing: `--ckpt.interval 10`, resume with `--ckpt.resume-step 10`; HF-compatible weights land at `outputs/weights/step_N`.

**Install** (Linux; NVIDIA only): `curl -sSL https://raw.githubusercontent.com/PrimeIntellect-ai/prime-rl/main/scripts/install.sh | bash`, or clone + `git submodule update --init -- deps/verifiers deps/renderers deps/research-environments deps/pydantic-config` + `uv sync --all-extras`.

**Hardware floor**: ≥1 NVIDIA GPU (3090/4090/5090/A100/H100/H200/B200). Single-GPU debug runs supported; realistic minimal RL = **2 GPUs** (1 inference + 1 trainer). Canonical small examples: `reverse_text` (Qwen3-0.6B, minutes on a consumer GPU), `alphabet_sort` (Qwen3-4B LoRA, one H100 ≈1 h), `wordle` (Qwen3-1.7B, 2–4 H100s). Scales to 1000+ GPUs / 1T+ MoE via one-line SLURM or Kubernetes/Helm; advanced examples go to 2048 GPUs.

## 2. Verifiers + Environments Hub

**`verifiers`** (https://github.com/PrimeIntellect-ai/verifiers, MIT) — created by Will Brown, now a Prime Intellect org project; it is *the* env SDK for the whole stack. PyPI `verifiers`; README news tops out at v0.1.14 (May 2026) but prime-rl 0.7.0 requires `>=0.2.0`, so a 0.2.x line exists — pin exactly. A **v1 Taskset/Harness API** shipped in 2026 alongside the legacy v0 env classes (v0 still works; most Hub envs use it). Docs: https://verifiers.readthedocs.io and https://docs.primeintellect.ai/verifiers/v1/overview.md.

**An environment = dataset + harness + rubric**, packaged as a self-contained Python module exposing `load_environment(**args)`. **Class hierarchy** (tech report §2.2.1): `Environment` → `SingleTurnEnv` (one model response); `MultiTurnEnv` (`env_response()`, `is_completed()`, per-rollout wall-clock timeouts since v0.1.13) → `ToolEnv` (native OpenAI-format tool calling from plain Python functions) → `StatefulToolEnv` (state-dependent tool args, e.g. sandbox IDs) → `SandboxEnv` (containerized execution) → `CodeEnv` (runs test cases against generated code — the `i3-code` pattern). `EnvGroup` concatenates envs with a task-ID routing column — this is how INTELLECT-3 trained on many envs simultaneously.

**Rubric API**: `vf.Rubric(funcs=[...], weights=[...])`; each reward function receives `(prompt, completion, answer, state)` (plus injected objects via `class_objects`) and returns a scalar; **async reward functions are first-class**; final reward = weighted sum. `vf.RubricGroup([rubricA, rubricB])` composes rubrics. `vf.JudgeRubric(judge_prompt=..., judge_client=AsyncOpenAI(...))` for LLM-as-judge — any OpenAI-compatible endpoint; judge responses are cached in `state` (defaults `parallelize_scoring=False` so the cache is hit). `vf.ToolRubric` for tool-usage metrics/penalties. Parsers: `XMLParser`, `ThinkParser`, custom subclasses. Per-group scoring (voting/ranking across samples of one prompt) is supported by overriding the scoring interface.

**Environments Hub** (https://hub.primeintellect.ai): envs are versioned Python wheels (pyproject + hatchling, pinned deps, standardized entry point). CLI flow:
```
uv tool install prime && prime login
prime env init my-env                 # template in ./environments/my_env
prime env install owner/name[@0.1.0]  # or: uv add <name> --index https://hub.primeintellect.ai/owner/name
prime eval run my-env -m openai/gpt-5-nano   # hosted eval; vf-eval for local
prime env push --path ./environments/my_env  # publish
```
Hub scale claim: 1,000+ envs / 250+ creators / 100k+ downloads (May 2026, community-sourced — treat as approximate). Vetted training-ready envs live in the `prime-environments` repo and the `research-environments` submodule (includes `primeintellect/i3-math`, `i3-code`, `i3-science`, `i3-logic`, `deepdive`, `deepswe`, `mini-swe-agent-plus`, plus gsm8k/math/wordle/alphabet-sort templates).

**How prime-rl consumes an env**: the orchestrator imports the installed module by `id`, calls `load_environment(**args)`, and gets back rollouts with rewards, token IDs, logprobs (straight from vLLM), and attention masks. Zero trainer-side code.

## 3. Reward design patterns for code generation

- **Execution-verified rewards are the house pattern.** `i3-code`: single-turn Python challenges, up to **15 test cases per problem executed inside Prime Sandboxes, 4,000+ concurrent sandboxes** during training; **on any sandbox failure the completion is masked out** (no reward hacking via infra errors). SWE envs run repo test suites (fail→pass) with the same masking rule.
- **Sandboxes**: the recommended path is **Prime Sandboxes** (managed; gVisor/`runsc` isolation, configurable network policy, warm pools, image streaming, <10 s cold start for arbitrary images, near-instant for pre-warmed; 256 sandboxes/node bin-packing; Rust gateway bypasses the k8s API server — naive `kubectl exec` orchestration hit 2.5 s/command latency at scale). Pricing: **$0.05/CPU-core/hr + $0.01/GB-RAM/hr + $0.001/GB-disk/hr** (1 core/2 GB/10 GB ≈ $0.08/hr); **CPU-only today** (GPU sandboxes on roadmap); default cap 512 active sandboxes. SDK + `prime sandbox create/run` CLI (https://docs.primeintellect.ai/sandboxes/overview.md). For local dev, a subprocess with timeouts works; e2b/Modal are usable via a custom `SandboxEnv` backend but are *not* first-class integrations.
- **Judge integration**: `JudgeRubric` with any endpoint; INTELLECT-3's math/science pattern is rule-based verify (`math-verify`) **plus** an LLM judge (`opencompass/CompassVerifier-7B`) only to re-check rule-rejected answers — kills false negatives without judge-everything cost.
- **Combining components**: weighted `Rubric(funcs, weights)` or `RubricGroup`; keep binary verifiable signals dominant and judge/style terms small (e.g. 0.1–0.3 weight).
- **Documented anti-reward-hacking measures**: `gibberish` (mean-logprob floor) / `repetition` (n-gram) / `zero_advantage` filters; `length_penalty` on GRPO; DPPO token masks + one-sided ratio clamp; INTELLECT-2 used two-sided GRPO clipping + aggressive grad-norm clipping for 32B stability; **online data filtering** — difficulty pools (easy/normal/hard by observed solve rate) and discarding all-pass/all-fail groups so batches keep learning signal.

## 4. Data / rollout pipeline

- **Datasets**: HF `datasets` rows with at least `prompt` (+ `answer`/`info` metadata); passed via env `args` (e.g. `args = { dataset_name = "openai/gsm8k" }`). Multiple envs mix via `ratio`; same env can repeat with different args (set distinct `name`).
- **Generation config**: `[orchestrator.sampling]` (temperature, `max_tokens`), `group_size` per env, `batch_size` (rollouts per step). INTELLECT-3 reference points: group size 8–16, batch 256–512; DeepDive ablation: group 16, batch 512.
- **Staleness control**: fixed one-step overlap by default; `max_off_policy_steps` discards over-stale rollouts; DPPO importance-ratio clamp + KL-to-sampling-policy term correct the residual off-policyness. Rollouts are token-in/token-out via `renderers` (exact-prefix invariant across turns; hand-coded renderers ship for qwen3, qwen3.5, glm4.5, glm5, **kimi-k2, kimi-k2.5**, deepseek-v3, minimax-m2, gpt-oss, nemotron-3 — anything else falls back to a generic chat-template renderer with retokenization risk). Multi-turn trajectories merge by best-effort interleaving (extension property; graceful per-sample fallback).
- **Eval loop**: online evals interleave into the same inference pool (overhead hidden); offline via `prime eval run` / `vf-eval`; `[[orchestrator.eval.env]]` in config.

## 6. Practical recipe — Manim scene-generation RL

**Base model.** Start with **Qwen3-4B-Instruct-2507** or Qwen3-8B (dense, HF-fallback trainer path, runs on 2 GPUs; LoRA on 1×H100). Step up to **Qwen3-30B-A3B** (optimized MoE path) for quality. **Kimi K2 Thinking**: open weights on HF (`moonshotai/Kimi-K2-Thinking`, 1T total/32B active MoE, native INT4 QAT, vLLM/SGLang supported) and prime-rl ships `kimi-k2`/`kimi-k2.5` renderers — **but** the K2 architecture is *not* in prime-rl's optimized trainer family table (no EP/CP custom code; generic `impl="hf"` path only), and 1T params needs 100s of GPUs. Verdict: **don't train K2 for a first experiment — use K2 Thinking (API) as the teacher (OPD) and/or judge**, and train a Qwen student. That also fits Hosted Training's per-token economics.

**Environment skeleton** (v0 API, `prime env init manim-env`):
```
environments/manim_env/
├── manim_env.py      # load_environment(**args)
├── pyproject.toml    # deps: verifiers; dataset from HF
└── README.md
```
In `manim_env.py`: `SingleTurnEnv` + HF dataset (`prompt` = scene spec, `info` = tags/difficulty) + custom parser (extract last python code block, `compile()` check) + `Rubric(funcs=[compiles, renders, visual_quality], weights=[0.2, 0.5, 0.3])`:
- `compiles`: AST/`py_compile` + contains a `Scene` subclass → cheap, async.
- `renders`: run `manim render -ql --media_dir <tmp> scene.py SceneName` with a hard timeout (~120 s) — subprocess locally for dev; **Prime Sandbox with a pre-built manim+ffmpeg(+LaTeX) image** for training scale. Mask (don't score) on infra failure, exactly like `i3-code`.
- `visual_quality`: `JudgeRubric` with a VLM endpoint scoring sampled frames (flag: verify image-input support for your judge path; otherwise use heuristic proxies — frame variance, non-black-pixel ratio, video duration, object count — which are safer against hacking anyway).
- Optional multi-turn upgrade: feed render stderr back via `env_response` (MultiTurnEnv, `max_turns=3`) — compile-fix loops are where RL shines; per-rollout wall-clock timeouts exist for this.

**Wire-up**: `prime env install` locally → smoke-test with `prime eval run manim-env -m <api-model>` → `rl.toml`:
```toml
[[orchestrator.train.env]]
id = "manim-env"
group_size = 8            # rollouts per prompt
[orchestrator.sampling]
max_tokens = 4096
[orchestrator.algo]
type = "grpo"
[orchestrator.algo.length_penalty]
type = "linear"
```
`uv run rl @ rl.toml` on 2 GPUs (e.g. 2×4090 for 4B; 2×A100/H100 for 8–14B). Do an SFT warmup on K2-Thinking-generated, render-verified traces (the SYNTHETIC-1/2 pattern) before GRPO — cold-start RL on sparse render rewards is slow.

**Known pitfalls**: (1) **render latency vs throughput** — train at `-ql`/low fps, size the orchestrator's rollout pool so rendering overlaps generation; async design tolerates the stragglers. (2) **Reward variance/sparsity** — shaped component weights + difficulty pools (drop all-pass/all-fail groups); group baseline handles the rest. (3) **Reward hacking** — empty/black scenes that "render": enforce heuristics + judge, `zero_advantage`/`repetition`/`gibberish` filters, length penalty. (4) **Sandbox safety** — never exec model code on the host; gVisor + no-network sandbox; LaTeX in images inflates cold-start (pre-bake it). (5) **Retokenization drift** in multi-turn — rely on the shipped renderers, don't re-render history. (6) Version churn — pin `prime-rl`/`verifiers` exactly; the v0→v1 API transition is still settling.
